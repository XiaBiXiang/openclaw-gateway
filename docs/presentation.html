Local First / Hybrid Routing / OpenAI Compatible

让 本地模型 和 云端模型 在一个入口里协同工作

OpenClaw Gateway 不是又一个聊天壳子，而是一个放在客户端与模型之间的推理路由层。它负责在隐私、复杂度、时延和稳定性之间做出可解释的选择。

Project Lens

2 对外同时兼容 `chat/completions` 与 `responses` 两类接口
1 对上层只暴露一个稳定网关入口，降低集成复杂度
∞ 本地与云端 provider 可继续扩展，不锁死具体模型厂商

The Tension

真正难的不是接模型，而是什么时候该切换

  • 只用本地模型 隐私更好，成本更可控，但复杂任务、长上下文和高质量推理很容易顶到上限。
  • 只用云端模型 能力更强，但隐私、离线能力、响应时延和持续成本都会成为产品瓶颈。
  • 客户端自己判断 每个客户端都要重复实现策略逻辑，协议碎片化，后续想做观测、治理和策略发布会非常痛苦。

Single Entry Strategy

把决策从客户端拿出来，集中到一个网关层

Client OpenClaw / App / Service
→
Gateway 策略判断 + 协议转换
→
Local MLX / OpenAI-compatible
或
Cloud OpenAI-compatible / Responses
Client Layer 上层只看见一个入口

OpenClaw 或其他客户端不再感知背后到底有几个 provider，也不需要知道何时走本地、何时走云端。

Policy Layer 网关负责做选择

网关根据隐私、离线、复杂度、上下文长度和历史会话状态决定是否升级到云端。

Provider Layer 本地与云端继续解耦

本地 provider 和云端 provider 都通过抽象适配层接入，后面可以平滑替换具体模型来源。

Routing Signals

策略不是拍脑袋，先做可解释路由

首版不追求复杂模型裁决器，而是先把影响实际效果的关键信号纳入同一个判断面。

Privacy / Sensitive

高隐私、高敏感场景直接强制本地，优先守住数据边界。

Offline

离线模式下允许强制本地，保障设备脱网时仍能继续工作。

Complexity

复杂任务越过阈值后自动升级到云端，避免小模型硬扛高难请求。

Context Limit

根据估算 token 数和本地上下文上限做预判，减少本地端无意义失败。

Stability Layer

Sticky Session: 防止路由来回抖动
Cloud Dwell Time: 上云后保持一段时间
Decision Endpoint: 可直接查看本次为什么这么选

Protocol Boundary

同一个网关，同时说两种OpenAI 方言

这层协议兼容非常关键。OpenClaw 偏 `responses`，很多本地服务偏 `chat/completions`，如果没有翻译层，端云协同会被接口差异卡死。

# Client -> Gateway POST /v1/responses POST /v1/chat/completions # Gateway -> Local POST /v1/chat/completions # Gateway -> Cloud POST /v1/responses # Normalization HTML upstream errors -> structured JSON errors 502/503/504 -> bounded retry
  • 对上层应用更稳定 客户端只需要适配一次 Gateway，不必为本地服务和云端服务分别实现不同的请求格式。
  • 对后续扩展更友好 后面如果要接更多本地引擎或更多云端模型厂商，改 provider 适配器比改所有客户端便宜得多。

Deployment Paths

当前已经具备三种落地形态

源码运行

适合开发调试和改策略。直接用 Go 跑起来，最快验证路由逻辑。

go run ./cmd/gateway -config /path/to/config.json
本地二进制

适合做个人环境稳定运行。构建后可直接作为本地网关进程使用。

go build -o bin/openclaw-gateway ./cmd/gateway
MLX + Gateway

适合你现在的方向。一个本地模型服务，加一个自动路由网关。

MLX_MODEL=mlx-community/Qwen3-4B-Instruct-2507-4bit ./scripts/run-hybrid-stack.sh

OpenClaw Integration

OpenClaw 只要指向 `http://127.0.0.1:18080/v1` 这一层即可。后面到底走 MLX 本地模型还是云端模型，由 Gateway 自动裁决。

Roadmap

先做出能跑的内核，再补控制面

Milestone 1: 可用网关

健康检查、路由决策、本地和云端 provider、OpenAI 兼容请求转发，先把端到端链路打通。

Milestone 2: 可用于真实场景

重试、超时、粘性会话、错误规整和观测能力继续补强，让它不只是 demo，而是能承压的中间层。

Milestone 3: 可发布

补齐 Docker、GitHub Actions、Release、安装文档和开源治理，让别人能直接拉下来用。

Milestone 4: Dashboard

再进入控制面，包括策略编辑器、历史查看、多实例管理和策略发布，这时 dashboard 才真正有意义。

Closing

这不是一个“再包一层 API”的项目，而是你后面做智能决策产品的执行底座

如果后面你要做自动路由决策、端云协同、Dashboard、多实例治理，OpenClaw Gateway 现在这层就是必须先打稳的基础层。先把统一入口、可解释路由和 provider 抽象做对，后续产品才能往上长。
